# Day 13 — LINQ Fundamentals: Declarative Queries over Collections (C# / .NET)

Objectives
- Learn declarative querying: express *what* you want, not *how* to iterate.
- Core LINQ ideas: projection, filtering, ordering, grouping, flattening, and joins.
- Map imperative iteration to declarative query stages in plain English.
- Understand deferred execution, materialization, and simple performance considerations.

Overview — declarative vs. imperative
Imperative code instructs the runtime step-by-step (loop, mutate state). Declarative querying describes the data transformation pipeline (filter → project → order → group). LINQ (Language Integrated Query) makes these pipelines concise, composable, and easier to reason about.

LINQ "personalities"
- Method (fluent) syntax: `collection.Where(...).Select(...).OrderBy(...)`
- Query (SQL-like) syntax: `from c in customers where ... select ...`
Both compile to the same underlying operations for LINQ to Objects; choose the style that improves readability for the task.

Sample domain types
```csharp
record Product(int Id, string Name, decimal Price);
record OrderItem(int ProductId, int Quantity);
record Order(int Id, int CustomerId, DateTime Date, List<OrderItem> Items);
```

Projection (Select)
Projection transforms each element into another shape (primitive, anonymous type, or DTO).
```csharp
var names = products.Select(p => p.Name).ToList();
var cheapProducts = products
    .Where(p => p.Price < 50m)
    .Select(p => new { p.Id, p.Name })
    .ToList();
```
In plain English: "From products, keep those costing less than $50, and select their id and name."

Filtering (Where)
`Where` reduces a sequence to elements that satisfy a predicate.
```csharp
var recentOrders = orders.Where(o => o.Date >= DateTime.UtcNow.AddDays(-30)).ToList();
```
In plain English: "From all orders, keep only those placed in the last 30 days."

Ordering (OrderBy / ThenBy)
Sort sequences by one or more keys.
```csharp
var topProducts = products
    .OrderByDescending(p => p.Price)
    .ThenBy(p => p.Name)
    .Take(5)
    .ToList();
```
In plain English: "Sort products by price descending, then by name, and take the first five."

Flattening (SelectMany)
Use `SelectMany` to flatten nested sequences (e.g., items inside orders).
```csharp
var allItems = orders.SelectMany(o => o.Items);
```
Imperative → declarative mapping:  
Imperative: "For each order, for each item, process item."  
Declarative: "Flatten all order items into a single sequence and process them."

Grouping & Aggregation (GroupBy, Sum, Count)
Group items by a key and compute aggregates per group.
```csharp
var totalsByProduct = orders
    .SelectMany(o => o.Items)
    .GroupBy(it => it.ProductId)
    .Select(g => new { ProductId = g.Key, TotalQty = g.Sum(it => it.Quantity) })
    .ToList();
```
Plain English: "Flatten order items, group them by product id, then for each group compute the sum of quantities."

Joins (Join, GroupJoin)
Join relates two sequences by keys. For LINQ-to-Objects `Join` is an inner join.
```csharp
var productSales = from p in products
                   join t in totalsByProduct on p.Id equals t.ProductId
                   select new { p.Name, t.TotalQty };
```
GroupJoin can serve as a left-join equivalent.

Lookups (ToLookup, ToDictionary)
Materialize keyed structures for repeated fast lookup:
- `ToLookup(keySelector)` produces a multi-valued map.
- `ToDictionary(keySelector, elementSelector)` produces a single-valued map.

Example:
```csharp
var itemsByProduct = orders.SelectMany(o => o.Items).ToLookup(i => i.ProductId);
```

Deferred execution and materialization
Most LINQ operators are deferred — they build a pipeline that runs only when enumerated. Materialize with `ToList()`, `ToArray()`, or call immediate operators (`Count()`, `First()`, `Any()`) when you need results now.

- Deferred: `Where`, `Select`, `SelectMany`, `OrderBy`, `GroupBy` (build pipeline).
- Immediate: `ToList`, `ToArray`, `Count`, `First`, `Any`, `Aggregate`.

Pitfall example:
```csharp
var expensive = items.Where(i => ExpensiveCheck(i)); // no work yet
var first = expensive.First(); // now work runs until first match
```
Be mindful of side effects and repeated enumeration (cache with `ToList()` if you iterate multiple times).

LINQ query stages in plain English (guided exercise examples)
- Filter customers active in last 90 days, project id+name, sort by most recent purchase:  
  "From customers, keep those with purchases in the last 90 days; select Id and Name; order by last purchase date descending."
- Compute top-5 products by revenue:  
  "Flatten all order items, join each to its product price, compute item revenue (quantity * price), group by product, sum revenue, order descending, take 5."
- Build word → document index:  
  "For each document, split into words, project pairs (word, docId), group by word and collect distinct docIds."

LINQ to Objects vs LINQ to Providers
LINQ to Objects executes in-memory delegates. LINQ to Providers (e.g., Entity Framework) translates expression trees to SQL or other backends — not all .NET methods are translatable. When targeting a DB:
- Keep expressions simple and provider-friendly.
- Prefer `IQueryable<T>` queries that the provider can translate.
- Materialize only when necessary (beware of client-side evaluation).

Performance tips (conceptual)
- Avoid repeated enumeration: cache results if you need multiple passes (`ToList()`).
- Avoid `Count()` inside loops on `IEnumerable` (it may re-enumerate).
- Use `ToDictionary`/`ToLookup` for repeated keyed access instead of repeated `Where`.
- LINQ expresses intent; if a pipeline is a performance hotspot, measure with BenchmarkDotNet and consider imperative optimization (or `Span<T>` for memory-critical loops).
- `OrderBy` sorts the whole sequence; if you need top-k, use `.OrderByDescending(...).Take(k)` or a heap-based approach for large inputs.

Best practices
- Use query syntax for complex multi-join/group queries when it improves clarity, otherwise prefer fluent method syntax.
- Prefer small, composable LINQ stages (each with a single responsibility).
- Materialize when you need a snapshot or multiple traversals.
- For DB-backed queries, design expressions that translate to SQL; test generated SQL when unsure.
